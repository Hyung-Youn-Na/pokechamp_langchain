## 2. 핵심 발견: 도구 사용 패턴 변화가 승률을 결정

### EXP-030 → EXP-031의 결정적 차이

EXP-031에서 `simulate_turn` 사용이 **64.7% → 19.0%** 로 대폭 감소하고, `calculate_damage`가 **23.2% → 55.3%** 로 주도권을 잡았습니다. 이 변화는 코드 수정이 아닌 **런타임 조건의 차이**로 발생한 것으로 보입니다 (같은 코드, 같은 프롬프트).

### 가설: `simulate_turn` 과의존이 EXP-030의 핵심 패인

1. **EXP-030에서 simulate_turn은 6,855회 호출** — 배틀당 229회
   - simulate_turn은 상대 행동을 추정해야 하므로 불확실성이 높음
   - 특히 중간 배틀(15-24턴)에서 상대의 교체/전략 변경이 빈번 → 시뮬레이션 오답 양산
   - **중간 배틀 승률 10%** 의 직접적 원인으로 추정

2. **EXP-031에서 calculate_damage가 주도** — 배틀당 98회
   - calculate_damage는 상대 행동 추정 없이 자체 기술의 데미지만 계산
   - 불확실성이 낮고, 결정에 더 직접적인 정보 제공
   - 승률 향상의 주요 요인으로 추정

3. **전체 도구 사용량 감소가 효율성 향상**
   - EXP-030: 턴당 14회 도구 호출 (과도한 분석)
   - EXP-031: 턴당 5회 도구 호출 (적절)
   - "분석 마비(analysis paralysis)"가 EXP-030의 문제였을 가능성

### 여전히 해결되지 않은 문제

- `get_team_analysis`는 EXP-031에서도 1회만 사용 — 팀 수준 전략 분석 부재
- `analyze_matchup`은 2.4%로 여전히 낮음 — 교체 결정의 질이 여전히 우려됨
